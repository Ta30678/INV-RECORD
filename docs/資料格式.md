# 資料格式

## 交易紀錄 frontmatter

一筆交易一個 `.md`，放在交易紀錄資料夾（預設 `40-交易紀錄/`）。
檔名慣例：`YYYY-MM-DD 代號 買進/賣出.md`（同日多筆自動加序號）。

```yaml
---
type: trade            # 必填，固定 trade；外掛只認這個值
date: 2026-07-01       # 必填，YYYY-MM-DD（交易日）
time: "09:05"          # 選填，成交時間 HH:mm（24 小時制、台灣時間）；務必加引號
seq: 1                 # 選填，同日多筆的排序（小的先處理）；有 time 時以 time 為準
ticker: "2330"         # 必填；務必加引號，避免 YAML 轉成數字
name: 台積電            # 選填，顯示名稱；缺少時用代號
action: buy            # 必填，buy 或 sell
qty: 1000              # 必填，股數（正數，支援零股）
price: 985             # 必填，成交價
fee: 1403              # 選填，手續費（元），預設 0
tax: 0                 # 選填，證交稅（元），賣出才有，預設 0
stock: "[[台積電 2330]]"  # 選填，連到個股筆記（Graph 邊）
theme: "[[AI 半導體]]"    # 選填，連到題材筆記
created: "2026-07-01 09:10"  # 建議填寫，本筆記建立時間（純記錄用，不影響計算）
updated: ""                  # 選填，重大修訂論點時手動填寫
---
```

驗證規則：欄位不合法時該筆**不列入計算**，並在儀表板「資料警告」區列出原因。
`time` 例外：格式錯誤時只忽略該欄（視同缺少），不影響整筆交易是否列入計算。
`created`/`updated` 純供人閱讀，外掛不會驗證也不影響計算——缺少不會產生警告。

## 筆記層級 frontmatter

| 層級 | type | 慣例欄位 | 連結欄位 |
|------|------|---------|---------|
| 總經 | `macro` | `date`、`created` | `up:` → 總經 MOC 或相關筆記 |
| 題材 | `theme` | `created` | `up:` → 總經筆記（選填） |
| 個股 | `stock` | `ticker`（加引號）、`created` | `up:` → 題材筆記（可一或多個） |
| 交易 | `trade` | 見上表（唯一由外掛實際驗證的一層） | `stock:`、`theme:` |

除交易筆記的欄位會被外掛剖析、驗證外，總經／題材／個股都不會被外掛剖析內容，
「慣例欄位」只是建議的寫作慣例（幫助自己複盤），不影響任何計算或警告。

`updated` 在四種筆記皆為選填欄位，外掛不會自動維護（避免每次修錯字都刷新，
淹沒真正的論點修訂日）；由你在重大改寫投資邏輯時手動填寫，格式與 `created` 相同。

## 時間欄位慣例

- **時區**：所有日期時間一律以**台灣時間（UTC+8）**為準，全案由外掛的 `utils/time.ts`
  單一模組產生，不會有時區不一致的問題。
- **格式**：`created`/`updated` 為帶引號字串 `"YYYY-MM-DD HH:mm"`（24 小時制，分鐘精度），
  刻意不用 ISO 8601 帶 `T`/時區的完整格式——YAML 會把裸 datetime 解析成 Date 物件並引入
  額外的時區轉換風險，帶引號的純字串在 Obsidian properties 裡最穩定、對人最可讀。
- `date`（交易日/總經事件日）與 `time`（成交時間）維持原有的日期／時間格式，不受影響。
- 用「新增交易紀錄」等指令建立的筆記，`created` 由外掛自動填入目前的台灣時間；
  用 90-模板 手動建立時，`created` 使用 Obsidian 內建的 Templates 外掛變數
  `{{date}} {{time}}`（預設格式已對應 `YYYY-MM-DD` 與 `HH:mm`）。

個股筆記的 `up:` 支援一檔個股橫跨多個題材：

```yaml
up: "[[AI 半導體]]"                              # 單一題材
up: ["[[AI 半導體]]", "[[先進封裝]]"]              # 多個題材（YAML 行內陣列）
```

交易筆記的 `theme:` 維持單一值——一筆交易應歸因到促成這次進出的主論點，強迫單選是交易紀律的一部分。

`up:`/`stock:`/`theme:` 只用於 Graph View 與反向連結（backlinks），不參與任何計算；
計算鏈只認交易筆記的 `ticker` 字串（見 FIFO 規則）。兩者原則上應該一致，
外掛在掃描時會唯讀比對：若 `stock:` 連結能解析到個股筆記、但該筆記的 `ticker`
與本筆交易的 `ticker` 不同，會在儀表板「資料警告」區列出，提醒你其中一邊可能改錯了
（外掛不會自動修正，只負責讓不一致變成看得見）。

## FIFO 損益規則

1. 所有交易依 **date → time → seq → 買進優先於賣出 → 檔名** 排序後依序處理
   （`time` 缺少時視為當日最晚；同鍵時買進先於賣出，取代舊版仰賴檔名字典序
   「買」在「賣」之前恰好正確的隱性行為）
2. **買進**：建立一批持股（lot），手續費按股攤入成本
   - 每股成本 = (成交價 × 股數 + 手續費) ÷ 股數
3. **賣出**：從最舊的 lot 開始消耗（先進先出）
   - 淨收入 = 成交價 × 股數 − 手續費 − 證交稅
   - 已實現損益 = 淨收入 − 消耗掉的 lot 成本合計
4. **賣超**（賣出股數 > 持有股數）：不做空，只實現可覆蓋的部分，
   費稅按比例分攤，並在儀表板記警告
5. **不在 v1 範圍**：當沖證交稅減半、除權息成本調整、庫藏股

範例（vault 模板內建的兩筆交易）：

```
2026-05-05 買 1000 股 @960，手續費 1,368
  → 每股成本 = 961.368
2026-06-15 賣 500 股 @1000，手續費 713、證交稅 1,500
  → 淨收入 = 500,000 − 2,213 = 497,787
  → 成本   = 500 × 961.368 = 480,684
  → 已實現 = +17,103；剩 500 股（成本 961.368）繼續持有
```

## 期間報酬定義（v1.1）

績效口徑三分法，三種口徑各自獨立呈現、永不混算（沒有現金帳／出入金台帳、也不做
歷史 as-of 估值，這是誠實呈現的唯一方式）：

1. **期間已實現報酬率** = 範圍內已實現損益 ÷ 範圍內平倉成本——唯一受時間範圍
   （本月／上月／今年／近一年／全部）約束的百分比
2. **目前持倉未實現報酬率** = 未實現損益合計 ÷ 目前持倉成本合計——快照，永遠用
   最新報價計算，不受時間範圍影響
3. **總績效（成立以來）** = (累計已實現損益＋目前未實現損益) ÷ 累計買進投入成本
   （所有買進交易的成交金額＋手續費總和）——只在時間範圍選「全部」時才顯示，
   因為此時分子分母口徑天然一致；賣出後再買回會墊高分母，是刻意的保守估計

不做任何年化：短持有期外推成年率會產生誤導性極端值（例如持有 10 天賺 3% 外推
成年化近 200%）。改用**加權平均持有天數**（依 FIFO 消耗股數加權：
Σ 消耗股數 ×(賣出日−lot 買進日) ÷ 總消耗股數，日差以 UTC epoch-day 計算）補上
時間維度，呈現於「個股別已實現明細」表。

「個股別已實現明細」依 ticker 分桶，欄位：賣出筆數、已實現損益、報酬率
（= 該檔損益 ÷ 該檔平倉成本）、勝率（該檔 pnl>0 的賣出筆數 ÷ 該檔賣出筆數）、
加權平均持有天數；依已實現損益由大到小排序。

時間範圍五檔預設選項的起訖日期計算（皆含頭尾）：

| 選項 | 起訖 |
|------|------|
| 本月 | 當月 1 號 ～ 今天 |
| 上月 | 上個月 1 號 ～ 上個月最後一天（1 月的上月正確跨年到去年 12 月） |
| 今年 | 今年 1/1 ～ 今天（曆年，不是 rolling 12 個月） |
| 近一年 | 今天往前 364 天 ～ 今天（rolling 365 天，含頭尾） |
| 全部 | 不過濾 |

自訂起訖日期選擇器延至 v1.2；資料層的過濾函式已支援任意 from/to，UI 只是五組
預設語法糖。K 線圖的 `range`（`1mo`/`3mo`/…）是「圖表回看窗」，儀表板的時間範圍
是「會計期間」，兩者語意本質不同，刻意不共用詞彙。

## 費用試算（新增交易 Modal 用）

- 手續費 = 成交金額 × 費率（預設 0.1425%）× 折扣，最低 20 元，四捨五入到元
- 證交稅 = 賣出成交金額 × 稅率（預設 0.3%），四捨五入到元
- 試算值只是預填，成交回報的實際金額可直接改

## Yahoo 資料

- 端點：`https://query1.finance.yahoo.com/v8/finance/chart/{代號}.TW`
- K 線與即時報價共用此端點（`meta.regularMarketPrice`）
- Yahoo 時間戳為 UTC，外掛 +8 小時後對齊台灣交易日
- 整根 OHLC 為 null 的 K 棒（停牌/缺漏）會自動略過
- 快取：同一張圖在 TTL 內不重抓；Yahoo 連不上時顯示上次快取並標示警告
- 每次 `getChart`/`getQuote` 都會回傳 `fetchedAt`（抓取或命中快取的時間），
  UI 用它顯示「資料截至」「報價更新於」「抓取於」等時間戳（見使用說明的時間紀錄一節）
- `meta.regularMarketTime`（unix 秒）用來判斷目前顯示的是盤中報價還是收盤報價；
  取不到時不做假設，不顯示盤中/收盤後綴
